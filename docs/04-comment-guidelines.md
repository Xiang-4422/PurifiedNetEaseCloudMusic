# 04. 代码注释规范

## 1. 目标

本规范用于约束本项目后续新增、修改和重构代码时的注释写法。

注释的唯一目标是帮助后来者理解：

- 为什么这样实现
- 这段实现受什么约束
- 为什么不能直接改成更直观的写法
- 类型、方法、属性对外承担什么职责

所有 Dart public 成员必须有文档注释；方法体内部的局部实现注释仍按需编写。

## 2. 核心原则

### 2.1 注释优先解释原因，必要时补最小行为说明

注释必须优先说明以下内容之一：

- 设计意图
- 限制条件
- 取舍原因
- 潜在误用风险
- 性能或稳定性考虑
- 外部依赖或平台兼容背景

方法体内部默认不写“代码正在做什么”。

但以下场景允许用极少量注释说明行为：

- 公共 API、入口函数、核心类型，对外职责不够一眼看清时
- 跨模块边界，调用方需要先理解输入、输出或副作用时
- 返回值、序列化格式、状态机流转存在固定约定时

允许的行为说明必须满足：

- 只说明职责、输入输出或副作用
- 长度应明显短于代码本身
- 说明完行为后，优先补“为什么这样设计”

不允许退化成逐行翻译代码。

### 2.2 不确定就不写

如果无法确认某段方法体内部逻辑背后的真实原因：

- 宁可不写
- 不猜测作者意图
- 不补“看起来合理”的解释

声明级注释不能因为“不确定原因”而省略；至少要基于当前签名和调用边界说明职责、输入输出、状态含义或副作用。

### 2.3 public 成员必须注释，方法体内部少写但写准

- 所有 Dart public 成员必须使用文档注释说明职责、输入输出、状态含义或关键副作用
- `analysis_options.yaml` 已启用 `public_member_api_docs`，缺少 public 文档注释会在 `dart analyze` 中报出
- private 成员不由 `public_member_api_docs` 强制检查，但存在特殊约束、反常规实现或误用风险时仍应补充注释
- 方法体内部只在存在特殊约束、反常规实现或误用风险时补充注释
- 一段局部逻辑最多保留一个高价值注释
- 避免连续多行把代码翻译成人话
- 注释应尽量短，直接说明关键约束

## 3. 必须补充注释的情况

以下 public 声明必须补充文档注释：

- public 类、枚举、扩展、mixin、typedef
- public 方法、函数、构造函数、getter、setter
- public 字段、属性、顶层变量、常量
- domain entity、application service/usecase、repository、data source、DAO、controller 公开状态和跨层 port 中的 public 成员

本文中的 public 指 Dart 中非 `_` 开头的声明。项目接受注释数量增加，因此不再为 DTO、enum value、生命周期样板方法、override 方法或 widget helper 设置项目级豁免。

声明级注释的重点是说明职责和使用边界，允许用一句话说明“做什么”；如果存在重要约束，应继续说明“为什么”。

private 声明不被 `public_member_api_docs` 强制要求，但方法体内部或 private 成员存在以下情况时，优先补充局部注释：

- 非直观或反常规实现
- workaround、临时兼容方案、hack
- 与平台、框架、三方库或外部接口相关的特殊行为
- 生命周期、并发、缓存、性能、滚动同步、播放器联动等关键稳定性处理
- 可能被误删、误改、误用的逻辑
- 无法只靠命名完整表达原因的代码

## 4. 禁止的注释

以下方法体内部局部注释禁止出现：

- 单纯复述代码行为
- 没有信息量的描述
- 与代码重复表达同一件事
- 仅描述“做了什么”，不说明“为什么”

声明级注释允许用最小行为说明职责、输入输出、状态含义或副作用。

典型反例：

- `初始化...`
- `设置...`
- `调用...`
- `判断...`
- `用于处理...`
- `这里是...`
- `把 A 赋值给 B`
- `循环列表并处理每一项`
- `如果为空就返回`

## 5. 注释形式

### 5.1 单行注释优先

默认使用：

```dart
// 注释内容
```

适用于局部特殊处理、约束说明和实现原因。

### 5.2 文档注释使用场景

public 声明必须使用文档注释；private 声明按复杂度和误用风险补注释。

推荐使用：

```dart
/// 注释内容
```

## 6. 内容要求

### 6.1 默认使用中文

注释默认使用中文。

以下内容可保留英文原词：

- 库名
- 协议名
- 平台术语
- 外部系统术语

### 6.2 临时方案必须写清边界

如果是临时方案，尽量写清楚：

- 为什么现在只能这样写
- 它在规避什么问题
- 未来什么条件下可以替换

禁止只写：

- `临时处理`
- `hack`
- `兼容问题`

### 6.3 不描述理想状态，只描述当前约束

除非当前实现正是被某个限制逼出来的，否则不要在注释里描述未来理想结构。

注释应该绑定当前实现，而不是描述希望以后怎么写。

### 6.4 行为说明的边界

public 声明应解释“做什么”，方法体内部注释仍优先解释“为什么”。建议控制在以下范围：

- 类或模块：一句话说明职责
- 方法：一句话说明输入输出、关键副作用或调用约定
- 属性：一句话说明状态含义、单位、生命周期或是否可为空
- 局部逻辑：原则上仍只解释原因，不解释行为

如果方法体内部的一条注释删掉“为什么”之后只剩行为描述，通常说明这条注释还不够好。

## 7. 清理规则

后续重构时，以下注释默认应删除或改写：

- 已经过时的注释
- 与当前实现不一致的注释
- 只有 TODO 但没有实际决策价值的注释
- 低质量、低信息量、翻译代码式注释

## 8. 项目执行规则

从本规范生效起：

- 新增代码必须遵守本规范
- 所有新增 public 声明必须补齐文档注释
- 修改存量 public 声明时，应同步补齐该声明的文档注释
- 修改旧代码时，可顺手清理低质量注释
- `dart analyze` 必须通过，`public_member_api_docs` 失败视为不符合规范
- 方法体内部不为了“看起来完整”而补注释数量
- 注释质量优先级高于注释覆盖率

## 9. 评审标准

判断一条注释是否保留，可用以下标准：

- 不看注释也能看懂代码行为，但看了注释能理解为什么必须这样写：保留
- 不看注释时很难快速判断入口职责、输入输出或副作用，而一两句话能明显降低误用成本：保留
- 注释只是把代码再说一遍：删除
- 注释无法帮助后续维护者做修改决策：删除
- 注释能防止错误重构、错误删除、错误替换：保留
