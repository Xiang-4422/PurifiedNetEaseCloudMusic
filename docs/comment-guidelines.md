# 代码注释规范

## 1. 目标

本规范用于约束本项目后续新增、修改和重构代码时的注释写法。

注释的唯一目标是帮助后来者理解：

- 为什么这样实现
- 这段实现受什么约束
- 为什么不能直接改成更直观的写法

如果代码本身已经足够清晰，默认不写注释。

## 2. 核心原则

### 2.1 注释只解释原因，不复述行为

注释必须优先说明以下内容之一：

- 设计意图
- 限制条件
- 取舍原因
- 潜在误用风险
- 性能或稳定性考虑
- 外部依赖或平台兼容背景

禁止用注释复述代码正在做什么。

### 2.2 不确定就不写

如果无法确认某段逻辑背后的真实原因：

- 宁可不写
- 不猜测作者意图
- 不补“看起来合理”的解释

### 2.3 少写，但要写准

- 一段逻辑最多保留一个高价值注释
- 避免连续多行把代码翻译成人话
- 注释应尽量短，直接说明关键约束

## 3. 必须补充注释的情况

以下情况优先补充注释：

- 非直观或反常规实现
- workaround、临时兼容方案、hack
- 与平台、框架、三方库或外部接口相关的特殊行为
- 生命周期、并发、缓存、性能、滚动同步、播放器联动等关键稳定性处理
- 可能被误删、误改、误用的逻辑
- 无法只靠命名完整表达原因的代码

## 4. 禁止的注释

以下注释禁止出现：

- 单纯复述代码行为
- 没有信息量的描述
- 与代码重复表达同一件事
- 仅描述“做了什么”，不说明“为什么”

典型反例：

- `初始化...`
- `设置...`
- `调用...`
- `判断...`
- `用于处理...`
- `这里是...`

## 5. 注释形式

### 5.1 单行注释优先

默认使用：

```dart
// 注释内容
```

适用于局部特殊处理、约束说明和实现原因。

### 5.2 文档注释使用场景

函数、类、模块满足以下任一条件时，可使用文档注释：

- 是公共 API
- 跨模块调用
- 容易误用
- 存在重要前置约束

推荐使用：

```dart
/// 注释内容
```

## 6. 内容要求

### 6.1 默认使用中文

注释默认使用中文。

以下内容可保留英文原词：

- 库名
- 协议名
- 平台术语
- 外部系统术语

### 6.2 临时方案必须写清边界

如果是临时方案，尽量写清楚：

- 为什么现在只能这样写
- 它在规避什么问题
- 未来什么条件下可以替换

禁止只写：

- `临时处理`
- `hack`
- `兼容问题`

### 6.3 不描述理想状态，只描述当前约束

除非当前实现正是被某个限制逼出来的，否则不要在注释里描述未来理想结构。

注释应该绑定当前实现，而不是描述希望以后怎么写。

## 7. 清理规则

后续重构时，以下注释默认应删除或改写：

- 已经过时的注释
- 与当前实现不一致的注释
- 只有 TODO 但没有实际决策价值的注释
- 低质量、低信息量、翻译代码式注释

## 8. 项目执行规则

从本规范生效起：

- 新增代码必须遵守本规范
- 修改旧代码时，可顺手清理低质量注释
- 不为了“看起来完整”而补注释数量
- 注释质量优先级高于注释覆盖率

## 9. 评审标准

判断一条注释是否保留，可用以下标准：

- 不看注释也能看懂代码行为，但看了注释能理解为什么必须这样写：保留
- 注释只是把代码再说一遍：删除
- 注释无法帮助后续维护者做修改决策：删除
- 注释能防止错误重构、错误删除、错误替换：保留
