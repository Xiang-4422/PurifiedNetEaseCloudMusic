# 网易云接口开发包

## 1. 目标

把 `NeteaseCloudMusicApiEnhanced/api-enhanced` 复刻为可长期维护的 Dart SDK，并支持后续跟随上游刷新。

SDK 层只负责：

```text
接收 query -> 构造网易云内部请求 -> 选择加密方式 -> 发送请求 -> 返回原始响应
```

不负责 App 缓存、播放队列、UI、本地数据库和领域实体。

## 2. 上游来源

- 上游仓库：https://github.com/NeteaseCloudMusicApiEnhanced/api-enhanced
- 本地 submodule：`third_party/api-enhanced`
- Dart SDK：`packages/netease_music_api`

当前同步基线：

```text
版本：v4.36.1
commit：1454ee216ce9bb4133879bad333e4d2aed21c80b
module 数量：400
确认日期：2026-06-23
```

该基线由 `api_enhanced_raw_test.dart` 校验；刷新 submodule 后必须同步更新版本、commit 和 module 数量。

## 3. 刷新流程

```bash
git submodule update --remote third_party/api-enhanced
git diff --submodule

node packages/netease_music_api/tool/generate_api_enhanced_modules.js
node packages/netease_music_api/tool/generate_api_enhanced_modules.js --check
node packages/netease_music_api/tool/api_enhanced_coverage_report.js
node packages/netease_music_api/tool/api_enhanced_coverage_report.js --json
node packages/netease_music_api/tool/api_enhanced_coverage_report.js --markdown

flutter test packages/netease_music_api/test/api_enhanced_raw_test.dart
flutter test test/data/music_data/sources/netease/api_upstream_sync_test.dart
flutter test test/architecture
flutter analyze
```

首次拉取包含 submodule 的仓库后执行：

```bash
git submodule update --init --recursive
```

## 4. 生成器

生成器：

```text
packages/netease_music_api/tool/generate_api_enhanced_modules.js
```

生成产物：

```text
packages/netease_music_api/lib/src/generated/api_enhanced_modules.g.dart
packages/netease_music_api/lib/src/endpoints/raw/api_enhanced_raw_methods.g.dart
```

当前规则：

- 覆盖上游 `module/*.js`。
- 无 request path 的模块标为 `special`。
- 暂不支持的 crypto 标为 `special`。
- 生成产物记录上游版本和 submodule commit，确保上游内部逻辑变化也会触发 manifest 刷新。
- 生成器会格式化 Dart，并且只在内容变化时写回。
- `--check` 模式只校验生成产物是否最新，不写文件；产物落后时退出失败并提示重新生成。
- 生成器支持 `--generated-dir=` 和 `--raw-dir=` 输出到隔离目录，测试可在临时目录验证生成和 stale 检测，不需要改动工作区产物。

自动支持的 crypto：

```text
weapi
eapi
linuxapi
api
xeapi
```

`xeapi` 需要先完成 `register_xeapikey`，SDK 会持久化公钥状态并在请求响应后更新 xeapi session。

## 6. 当前覆盖基线

普通模块：

- 当前上游普通模块已全部进入 Node oracle fixture，缺口为 0。
- 测试会强制校验所有非 special module 都有 oracle fixture。
- `api_enhanced_coverage_report.js` 会输出当前覆盖报告和 limited 模块限制说明，并在普通模块缺 oracle、special 未归类、special oracle fixture 缺失或 limited 模块缺少限制说明时失败。
- coverage report 会输出 submodule 路径、当前 commit、dirty 状态、上游 module 文件数；当前 commit 读取失败、submodule dirty 或 generated manifest 与当前上游 `module/*.js` 不一致时都会失败。
- coverage report 会把 submodule commit 读取失败或 dirty 状态写入 `sdkDifferences`，避免上游跟随流程只读差异清单时漏掉 submodule 状态问题。
- coverage report 会校验 generated manifest 记录的上游版本和 commit 是否等于当前 submodule，避免刷新 submodule 后忘记重新生成 raw manifest。
- coverage report 会校验 generated manifest 中 module 和 raw method name 不重复，避免生成器异常导致 raw API 覆盖关系不清。
- coverage report 会校验 generated manifest 条目的 normal module path、crypto 和 HTTP method，避免生成器解析异常后继续产出不可调用 raw API。
- coverage report 会校验 generated manifest 的 list 与 `apiEnhancedModuleByName` map 一致，缺失、未知、重复 key 和字段漂移都会进入 `sdkDifferences`。
- coverage report 会校验 generated raw convenience methods 与 manifest 的 module/methodName 一致，缺失、未知、重复和命名漂移都会进入 `sdkDifferences`。
- coverage report 会校验 generated manifest list、manifest map 和 raw convenience methods 的排序稳定性，顺序漂移会进入 `sdkDifferences`。
- coverage report 会把 generated manifest 缺少上游模块或包含未知模块写入 `sdkDifferences`，避免上游刷新后只读差异清单时漏掉 raw 覆盖问题。
- coverage report 会校验 `api_enhanced_special_coverage.json` 的顶层字段、数组项、limited 原因、重复条目和排序稳定性，避免 special 覆盖状态配置写错或乱序后静默通过。
- coverage report 会校验每个 special module 都有 `requestModule` 手写分发，且分发 case 不能重复或指向非 special module，避免上游新增 special 后 raw dispatcher 漏实现。
- coverage report 支持 `--special-coverage=` 指向临时配置文件，raw 测试用它固定 special coverage 配置错误时必须失败。
- coverage report 支持 `--generated-manifest=` 指向临时 manifest，raw 测试用它固定 manifest 上游版本、commit 和模块集合错误时必须进入 `sdkDifferences`。
- coverage report 支持 `--raw-methods=` 指向临时 raw convenience methods，raw 测试用它固定 convenience methods 缺失、未知、重复和命名漂移必须失败。
- coverage report 支持 `--raw-dispatcher=` 指向临时 dispatcher，raw 测试用它固定 special dispatcher 缺失、未知和重复 case 必须失败。
- coverage report 的 `--json` 会输出 `sdkDifferences` 结构化差异清单，每个差异都带 `scope`，后续上游刷新和 SDK 收敛优先按该清单处理。
- coverage report 的 `--markdown` 会从同一份报告渲染人工复核摘要，包含上游基线、覆盖计数、逐个 special module 覆盖状态、special/runtime 状态和 `sdkDifferences` 差异表，避免手写差异清单与工具输出脱节。
- `sdkDifferences` 输出顺序按 module 名固定，raw 测试会阻断顺序漂移，避免上游刷新后产生无意义的差异清单噪音。
- special 覆盖配置引用不存在的 special module 时也会进入 `sdkDifferences`，避免自动化只读差异清单时漏掉上游对齐失败。
- coverage report 的 `--json` 会输出 `specialCoverageStatusByModule`，逐个 special module 标记覆盖类型、是否存在 Node oracle fixture，以及 limited 原因。
- coverage report 的 `--json` 会输出 `runtimeSupportedReasons`、`runtimeLimitedReasons` 和 `runtimeOptionStatusByName`，runtime option 层面的支持与限制必须可被工具读取，明确限制也必须进入 `sdkDifferences`。
- `api_enhanced_raw_test.dart` 会执行 coverage report 的 `--json` 模式，防止覆盖报告只停留在手动检查。
- `api_enhanced_raw_test.dart` 会在临时目录执行生成器，固定 `--check` 发现 stale 产物时失败且不会写回文件。
- 上游刷新后，如果新增普通模块没有 fixture，`api_enhanced_raw_test.dart` 必须失败。

special module 中，使用 Dart 本地行为测试固定：暂无。不能直接复刻上游的模块必须写入明确限制。

固定行为：

- `api` 动态 raw 请求的 uri、data、crypto、cookie option 已通过 Node oracle 固定；`data.cookie` 字符串会按上游同步为 request option cookie。
- `audio_match` 上游不走通用 request，而是直接请求音频指纹匹配地址；返回 envelope 已通过 no-request Node oracle 固定。
- `cloud` 按上游 songFile 完整流程固定 upload check、token alloc、cloud info、publish 四步请求；`needUpload=true` 的二进制上传仍通过 Dart 本地上传 helper 执行。
- `decrypt` 是本地纯解密 helper；`eapi`、`api`、`linuxapi`、`weapi` 响应、`xeapi` 响应以及不支持的请求解密 envelope 已通过 no-request Node oracle 固定。
- `eapi_decrypt` 是本地纯解密 helper；请求与响应解密 envelope 已通过 no-request Node oracle 固定。
- `inner_version` 上游不发请求并返回当前上游包版本，已通过 no-request Node oracle 固定。
- `login_qr_create` 的基础 PC 链接分支已通过 no-request Node oracle 固定；web chainId 和二维码图片分支保留 Dart 本地行为测试。
- `related_playlist` 上游直接返回 axios response 并追加 `body`；Dart raw envelope 已通过 no-request Node oracle 固定 `data` HTML 与解析结果。
- `register_anonimous` 上游内部生成 deviceId 并发送 xeapi 注册请求；测试用固定随机序列锁定 request metadata，Dart 保留 query 注入 deviceId 作为测试与 App 可控入口。
- `register_xeapikey` 上游内部生成 nonce/timestamp 并直接请求 axios；测试用固定时间和随机序列锁定 raw envelope，同时保留 Dart SDK 写入 xeapi public key 状态的副作用。
- `song_url_ncmget` 上游不发请求并固定返回空成功体，已通过 no-request Node oracle 固定。

明确限制：

- `cloud` 的四步请求序列已覆盖；Dart SDK 已解析常见 ID3v2、ID3v1、FLAC/Vorbis 和 MP4/M4A `ilst` 标签，ID3v2 与 FLAC/Vorbis 支持专辑艺术家 fallback，`filePath/tempFilePath` 只做有上限的文件前缀和 ID3v1 尾部探测，但仍未内置完整 `music-metadata` 等价能力，解析失败时使用 query 显式值或文件名 fallback。
- `scrobble_v1` 依赖上游 NCBL 加密、随机 multipart 构造和直接上传 helper；Dart SDK 固定返回包含 `module` 与 `unsupportedFeature` 的显式限制 envelope。
- `song_url_v1` 的 unblock 解灰成功路径依赖 `unblockmusic-utils`；Dart SDK 在 `unblock=true` 时降级到普通 xeapi 播放 URL 请求，基础 xeapi 请求和 `song_url_v1_302` fallback 已覆盖，source order 只属于未实现的解灰分支。
- `song_url_match` 依赖 `unblockmusic-utils`；Dart SDK 固定返回包含 `module` 与 `unsupportedFeature` 的显式限制 envelope。
- `proxy` 支持 HTTP/HTTPS 代理 URL，包含 `http://user:pass@host:port` 形式的基础认证代理；只有明确的 PAC 文件或 PAC scheme 暂不支持。

当前 SDK 差异清单：

- `cloud`：四步请求序列已覆盖；Dart 解析常见 ID3v2、ID3v1、FLAC/Vorbis 和 MP4/M4A ilst 标签，ID3v2 与 FLAC/Vorbis 支持专辑艺术家 fallback，但没有完整 music-metadata 等价能力，解析失败时使用 query 显式值或文件名 fallback。
- `scrobble_v1`：依赖上游 NCBL 加密、随机 multipart 构造和直接上传 helper；Dart SDK 当前固定返回包含 module 与 unsupportedFeature 的显式限制 envelope。
- `song_url_match`：依赖上游 unblockmusic-utils，Dart SDK 固定返回包含 module 与 unsupportedFeature 的显式限制 envelope。
- `song_url_v1`：基础 xeapi 和 302 fallback 已覆盖；unblock=true 会降级到普通 xeapi 播放 URL，实际解灰和 source order 仍依赖上游 unblockmusic-utils。
- `runtime:proxy.pac`：proxy 支持 HTTP/HTTPS 代理 URL 和基础认证；明确的 PAC 文件或 PAC scheme 暂不支持，Dart client 会显式抛 UnsupportedError。
- `runtime:source_order`：source order 只属于 song_url_v1 的 unblock 解灰路径；Dart SDK 当前未实现 unblockmusic-utils，普通 xeapi 播放 URL 请求会忽略 source。

当前 `sdkDifferences` 只包含 `cloud`、`scrobble_v1`、`song_url_match`、`song_url_v1`、`runtime:proxy.pac` 和 `runtime:source_order` 六个 limited 差异；如果后续新增缺失 oracle、未归类 special、缺少限制说明、runtime option 限制或 special coverage 配置结构错误，coverage report 会把它们作为差异项输出并失败。需要给自动化或后续 goal 消费完整结构时，使用 `api_enhanced_coverage_report.js --json`。

## 7. 运行时 option

- `realIP` 写入 `X-Real-IP` 和 `X-Forwarded-For`。
- `randomCNIP` 在 raw metadata 阶段生成请求 IP，显式 `realIP` 优先；SDK 运行时默认关闭，可通过 `NeteaseMusicApi.init(enableRandomCNIPByDefault: true)` 或 `configureRuntimeOptions` 全局开启，单次 query 的 `randomCNIP=false` 可以覆盖全局默认。
- `ua` 映射为原始 User-Agent。
- `checkToken` 写入 eapi header，并通过拦截后解密的 eapi payload 测试固定。
- `e_r` 进入 raw request options；eapi/weapi 请求开启后按加密响应读取并解密。
- `proxy` 映射为 Dart native `HttpClient` 代理规则，URL user info 会转为基础认证代理凭据。
- `domain`、`cookie` 进入 request options。

## 8. 测试要求

raw manifest 测试必须覆盖：

- manifest 覆盖所有上游 module。
- generated method name 唯一。
- normal module 能构建 request metadata。
- normal module 有 path。
- normal module crypto 受支持。
- special module 清单显式固定。
- special module 均归入 Node oracle、本地行为测试或明确限制。
- 明确限制必须有非空说明，说明为什么不能完整复刻以及当前替代行为。
- 覆盖报告 JSON 必须输出 `specialLimitedReasons` 和带 `scope` 的 `sdkDifferences`，让 SDK 差异清单可以被工具读取并按范围分派。
- 覆盖报告 JSON 必须输出 `upstreamCommit`、`upstreamDirty`、`upstreamModuleFileCount`、manifest 上游版本/commit 和 manifest 差异列表，保证上游刷新基线可追踪。
- 覆盖报告 JSON 必须输出 generated manifest 重复 module 和重复 method name 列表，保证 raw API 入口不会静默覆盖。
- 覆盖报告 JSON 必须输出 generated manifest 条目无效列表，保证 normal module 的 path、crypto 和 HTTP method 可被结构化检查。
- 覆盖报告 JSON 必须输出 generated manifest list/map 不一致列表，保证 `apiEnhancedModules` 和 `apiEnhancedModuleByName` 不会漂移。
- 覆盖报告 JSON 必须输出 generated raw convenience methods 的缺失、未知、重复和命名漂移列表，保证 raw 便利层完整跟随 manifest。
- 覆盖报告 JSON 必须输出 generated manifest list/map 和 raw convenience methods 的顺序漂移列表，保证生成器输出稳定可回归。
- 覆盖报告 JSON 必须输出 special coverage 配置无效项、重复项和顺序漂移列表，保证 special 状态配置可被测试固定。
- 覆盖报告 JSON 必须输出 special raw dispatcher 缺失、未知和重复 case 列表，保证手写 special 分发跟随 generated manifest。
- Node oracle fixture 覆盖所有 normal module。
- `xeapi` module 必须 special。
- pathless module 必须 special。
- unsupported crypto 不能静默 fallback。
- `randomCNIP` 必须生成可用 IPv4，不能覆盖显式 `realIP`，且 query 显式值必须能覆盖 SDK 默认值。
- `randomCNIP=false` 和 `e_r=true` 必须进入 Node oracle metadata 对齐。
- `checkToken` 必须进入 eapi 加密 payload 的 `header.X-antiCheatToken`。
- `proxy` 必须能映射为 Dart native proxy rule；基础认证代理必须解析凭据，普通代理主机或路径包含 `pac` 字样不能被误判为 PAC，不支持的代理形式必须显式报错。
- Node oracle fixture 必须验证 Dart metadata 与上游 Node request metadata 一致。

## 9. 后续 goal

1. 每次上游刷新后先跑生成器 check 和 raw 测试，确认普通模块 oracle 仍是 100%。
2. 上游新增 special module 时，优先归入 Node oracle；无法复刻时必须写明确限制，再更新文档。
3. 评估 `song_url_match` 和 unblock 替代方案。
4. 评估 PAC 代理是否需要补齐。
5. 评估 unblockmusic-utils 的 Dart 替代方案；在替代方案明确前，source order 只作为不支持的解灰能力记录。
