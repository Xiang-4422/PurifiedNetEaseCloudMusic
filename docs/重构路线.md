# 重构路线

## 1. 目标

后续重构目标是把当前应用收敛成一个自用、简约、快速、稳定的音乐播放器。

不做：

- 不推倒重来。
- 不迁移状态管理框架。
- 不重写播放器。
- 不把 App 缓存逻辑放进 SDK。
- 不让 UI 直接消费 SDK raw JSON。

## 2. 已完成方向

- 项目已从固定 application/usecase/port 分层收敛为 feature-first。
- 已删除大量只转发的薄层。
- 歌单详情已走本地优先。
- 自有业务 snapshot 持久化已移除。
- Drift 已成为业务事实来源。
- Hive 范围已收窄。
- `packages/netease_music_api` 已独立成 SDK 包。
- bootstrap 已从单个大文件拆成 UI、route、SDK、storage、数据源、repository、数据装配、播放、展示适配器、feature/controller 装配文件。
- 根路由实例、初始路由和登录失效导航已收口到 `route_bootstrap`，App 根组件只承接 `GetMaterialApp.router`。
- SDK 初始化和 `NeteaseMusicApi` 实例创建已收口到 `sdk_bootstrap`，数据装配只接收并注入已创建的 API 门面，`data/music_data/sources/netease` 不再保留远程初始化入口。
- Drift/Hive 初始化已收口到 `storage_bootstrap`，数据装配只消费已初始化的存储资源。
- 本地 data source、搜索/探索 cache store 和本地音乐来源已收口到 `data_source_bootstrap`。
- 核心 repository、feature repository 和远程 data source 创建/注册已收口到 `repository_bootstrap`，数据装配只保留启动顺序和恢复任务。
- `UserScopedDataSource` 已拆出用户资料、曲目列表、歌单列表、订阅状态、电台、同步标记等窄接口。
- `DriftUserScopedDataSource` 已退化为聚合器，真实表访问已拆到窄 Drift data source。
- 用户作用域窄 Drift data source 已只保留 DAO 委托，用户歌单列表和电台缓存表访问已下沉到 DAO。
- Drift 迁移和索引 SQL 已从 `drift_database.dart` 拆到 `schema/drift_schema_maintenance.dart`。
- Drift 表定义已从 `drift_database.dart` 拆出，并按 playback、local_resource、download、cache、library、playlist、user 业务域拆分。
- `PlaybackQueueItem` 已移除 `MediaItem` extras 和缓存 JSON 编解码职责，转换逻辑留在 playback adapter/codec。
- `PlaybackQueueItem.albumId`、`PlaybackQueueItem.artistIds`、`PlaybackQueueItem.sourceType`、`PlaybackQueueItem.localLyricsPath` 和 `PlaybackQueueItem.availability` 已字段化，UI 和服务层不再读取对应 metadata 动态键。
- 播放队列 MediaItem 和缓存 JSON 写出路径只使用显式字段；legacy metadata 只在读取旧缓存时迁移。
- 播放队列 mapper、MediaItem adapter 和缓存 codec 已过滤 `mv`、`fee`、`publishTime`、云盘和本地扫描来源等非播放 metadata，播放队列只保留自定义扩展。
- `Track.albumId` 和 `Track.artistIds` 已字段化，本地 DAO 使用专辑列和歌手关系表保存，旧 metadata 键只作为读写迁移输入。
- 登录标记、用户 session、设置项和图片主色缓存已通过 key-value 边界访问 Hive，`CacheBox` 全局入口不再暴露给对应存储实现。
- 旧 Hive 业务事实 key 已从 `app_cache_keys.dart` 移除，喜欢歌曲、用户歌单、推荐歌曲、下载任务、本地资源索引和播放恢复不再保留轻量存储入口。
- 搜索热词、探索分类和评论等短期 `app_cache_entries` key 已迁到 `AppCacheDataSource` 边界，feature cache store 不再依赖 `data/app_storage` key。
- 用户资料库和首页推荐的账号作用域本地数据加载已加入过期请求丢弃，旧账号异步返回不能覆盖当前账号状态。
- 喜欢歌曲队列加载会在落状态前复核账号、请求代次和喜欢歌曲 id 快照，旧账号或旧列表的慢返回不能覆盖当前队列。
- 喜欢歌曲列表远程重载成功后才替换当前列表，远程失败不会清空已展示的喜欢歌曲。
- 喜欢歌曲 id 刷新成功后会同步裁剪已加载的喜欢歌曲可见列表，并按新的 id 顺序保留仍有效的歌曲。
- 用户资料库启动刷新已改为远程快照整组成功后再更新可见状态，远程任一路失败时保留当前资料库状态。
- 用户资料库远程刷新成功但歌单为空时会同步清空可见喜欢入口和用户歌单，避免旧账号或旧列表继续残留。
- 后台登录态校验返回前会复核发起校验时的 `userId`，旧账号校验结果不能覆盖或清空新账号状态。
- 缓存登录恢复完成后会复位登录加载态，避免本地登录恢复成功但登录页仍停留在 loading。
- 后台登录态校验遇到网络异常时保留当前缓存 session，只有服务端明确返回未登录才清空本地账号状态。
- 只有登录标记但没有可用本地 session 时，会直接回退到二维码登录入口，不再凭 SDK cookie 远程恢复当前用户。
- 主动注销时本地账号 session 和登录标记会优先清理，远程 logout 只作为 SDK cookie/request session 清理尝试。
- 主动注销复用 `AuthController` 的登录页副作用通道，清理本地账号后回到登录页，UI 不直接操作登录路由。
- 二维码创建失败和轮询短暂异常会保留可重试状态，登录页不再因为没有二维码 URL 而空白。
- 用户资料页有本地资料时后台刷新失败会保留当前资料并挂载错误状态，首次加载失败时才进入纯错误态。
- 用户资料页刷新会丢弃旧代次结果，页面释放后慢返回的资料请求不能继续写入已释放状态。
- 本地资源索引保存和读取会验证文件真实存在，失效索引不会再进入播放资源包或本地歌曲列表。
- 播放源解析本地文件和网易云缓存文件前会二次确认文件存在，下载或缓存文件缺失时回退远程 URL，本地导入文件缺失时明确失败。
- 播放 URL 强制刷新结果只允许最新请求写入短缓存，旧的慢请求晚返回时不会覆盖已刷新的可用 URL。
- 播放恢复队列缓存按条容错，坏缓存项或恢复队列解码失败不会打断启动恢复，也不会把旧歌单上下文带回空队列。
- 播放队列确认底层歌曲时会校验歌曲 id 和索引一致，旧切源完成不能把新队列同索引歌曲误标为 confirmed。

- mini player 已补充展开播放器和播放/暂停的辅助语义与 Tooltip，收起态控制入口可被稳定识别。

- SDK coverage report 已检查 Node oracle 是否重复定义同一场景 fixture，避免 golden 覆盖关系不清。
- SDK coverage report 已检查 Node oracle fixture 结构，缺失 `module/query`、未知字段或控制字段类型错误都会阻断上游对齐流程。
- SDK coverage report 已校验 generated manifest 记录的上游版本和 commit，submodule 刷新后未重新生成 manifest 会直接暴露。
- SDK coverage report 已校验 generated manifest 中 module 和 raw method name 不重复，避免生成器异常导致 raw API 覆盖关系不清。

## 3. 架构整理

优先级：

1. 继续收紧登录、cookie、账号状态所有权。
2. 拆分 Drift 表、DAO 和 data source。
3. 继续审计 `Track.metadata` 中真正需要跨页面消费的业务字段；只有当前播放体验需要的字段才进入 `PlaybackQueueItem`。
4. 继续收紧 bootstrap 内部各装配文件的依赖方向。

## 4. 登录和账号

目标边界：

- SDK 管 cookie、token refresh、request session。
- App 管当前用户、账号作用域、本地缓存归属。

需要梳理：

- 冷启动恢复顺序。
- cookie 失效和刷新失败。
- 登出和切换账号。
- `user_*` 表和刷新 marker 的 `user_id` 隔离。
- 账号切换时，所有异步本地/远程刷新结果必须在落状态前复核当前 `userId`。

本地缓存登录必须同时具备登录标记和可解析、`userId` 非空的 session；损坏或匿名 session 不能把 App 带入首页。
登录页启动和后台校验只按完整 cached session 分支，孤立登录标记不能触发账号恢复；SDK cookie 仍只作为接口请求会话。
用户主动注销时，App 当前用户、账号作用域和本地登录标记必须立即清理；远程 logout 失败不能阻塞本地退出。
清理完成后必须通过 Auth 展示副作用回到登录页，避免用户停留在已登录壳层继续触发账号作用域页面。

## 5. 缓存策略

需要统一：

```text
歌曲详情      Drift 长期保存，远程刷新补全
歌单          元信息和曲目关系分开缓存
歌词          缺失拉取，成功后长期保存
封面          文件缓存和取色缓存分层
播放 URL      短 TTL 或内存缓存
喜欢歌曲      账号作用域
用户歌单      账号作用域
云盘          账号作用域
评论          短期缓存或后续建专表
搜索热词      短 TTL
```

## 6. 播放和下载

播放方向：

- 本地资源优先。
- 远程 URL 过期后刷新。
- 远程 URL 保留 query 签名参数，只在本地文件路径判断时裁剪 query。
- 播放恢复队列缓存已收紧为只保留明确本地路径，`file://` 会归一为本地文件路径，远程 URL 和其它 URI scheme 不进入恢复 session。
- 播放 URL 短 TTL、并发请求合并、本地资源优先重查和 LRU 淘汰收口在 `PlaybackUrlCacheCoordinator`，避免 `MusicDataRepository` 继续堆播放缓存状态。
- 远程替换失败时强制刷新 URL；高音质仍失败时降级到普通音质。
- SDK 不内置 unblockmusic-utils 替代链路；`song_url_v1` 的 `unblock=true` 会降级到普通播放 URL 请求，source order 不参与普通播放 URL 请求，App 播放链路负责失败后的刷新和音质降级。
- 本地导入文件播放时不在队列模型或播放源替换阶段追加播放缓存展示标记，避免把用户导入资源误当成播放缓存；正式下载和播放缓存资源仍可展示为已缓存。
- 下一首预取不阻塞当前播放。
- 相邻歌曲预取只在当前歌曲 confirmed 后后台启动，不能拖慢当前切歌和控制反馈。
- 恢复状态只保存 restore/session 语义。

下载方向：

- `download_tasks` 描述下载过程。
- `local_resource_entries` 描述最终可用资源。
- 播放只信任本地资源索引，不直接信任下载任务状态，也不能把本地曲目的 `sourceId` 当作隐式本地文件路径。
- 取消下载不能提前消费取消标记；已取消任务即使文件写入完成，也不能登记为本地资源。
- 本地资源索引只能代表真实存在的文件，读取或播放入口二次确认发现文件缺失时需要清理索引并回退到远程解析。
- 资源索引写入按本地导入、正式下载、播放缓存维护优先级。
- 已有播放缓存被正式下载命中时，只提升本地资源索引为正式下载事实并清理任务状态，不重新下载音频文件。
- 播放缓存清理只删除播放缓存来源资源，不能误删同一曲目的正式下载、本地导入、封面或歌词资源。

## 7. UI 方向

定位：自用、简约、纯粹听歌。

核心页面：

- 首页：最近播放、每日推荐、常用歌单。
- 资料库：喜欢、歌单、本地音乐、下载、云盘。
- 搜索：搜索框、热搜、历史、结果。
- 播放页：封面、歌曲信息、进度、歌词、队列、音质。
- 歌单页：封面、标题、操作区、歌曲列表。
- 设置页：账号、音质、缓存、下载、外观、调试。

避免：

- 信息流、复杂社交、运营 banner 成为主入口。
- 卡片套卡片。
- 过度模糊或取色影响可读性。
- 动效影响播放控制响应。

## 8. 性能指标

性能指标以 `AppPerformanceMetrics` 为准：

```text
key                       指标                 目标     事件
cold_start_interactive    冷启动到可交互       1500ms   app.startup.interactive
cached_playlist_open      打开已有缓存歌单     500ms    page.loadInitial.total
track_switch              切歌响应             300ms    switch.total
search_first_results      搜索首批结果         800ms    search.firstResults.total
mini_player_feedback      mini player 控制反馈 120ms    miniPlayer.feedback.total
```

所有 `elapsedMetric` 输出必须带上 `metric`、`target`、`status=ok/slow` 和剩余或超出预算，避免性能目标只停留在文档表格里。

搜索结果按歌曲、歌单、专辑、歌手分类独立发布，首个分类完成后立即更新状态，不能被最慢分类阻塞；远程搜索失败但已有本地结果时，先保留本地结果作为降级展示。

每次修改后的固定验证门槛：

```bash
flutter analyze
flutter test
git diff --check
```

这台开发机后续默认不执行 Android APK 构建和 adb 安装；只有用户单独要求或切换到更合适的构建环境时才补跑。

mini player 播放/暂停命令在播放 UI 命令服务层合并并发请求，避免快速重复点击打出多次底层播放命令。

mini player 播放/暂停反馈指标会在成功和失败路径都记录 `action` 与 `result`，底层命令异常不能让反馈耗时样本丢失。
